# Why a Rich Domain Model for Quote?

The old Quote model was an anemic model because it only stored data and did not protect its own rules. It had public properties, so any developer could directly change values like quote.Text = "" or create invalid objects using new Quote { Author = "" }. Validation rules were scattered in DTOs, endpoints, and repositories, which meant developers had to remember them manually. If someone forgot validation in one place, bad data could easily enter the system.
The rich model improves this by moving validation inside the Quote class itself. The only way to create a quote is through Quote.Create(author, text). This method checks if the author and text are valid before creating the object. Private setters stop outside code from changing values later and breaking the object state. Methods like SoftDelete() clearly define what changes are allowed.
A good example is a future bulk-import feature. A developer might directly create quotes and forget the author length validation. Since SQLite may not strictly enforce limits, invalid or oversized data could get stored and later returned to users. In the rich model, Quote.Create() rejects invalid input immediately, so the bad data never reaches the database or API responses.
